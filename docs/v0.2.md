# v0.2 发布说明

`v0.2.0` 在 v0.1 网关基础上补齐了 Codex OAuth 渠道、多协议规则路由和 Agent 统一网络出口。当前 `master` 对应 v0.2 系列。

## Codex OAuth Provider

- `codex` 与标准 `openai` API Key provider 隔离。
- 创建 Codex 端点时可以粘贴或上传 Auth JSON，也可以在 API Key 管理界面继续导入多个 JSON。
- 每个 Auth JSON 是一个独立 Key，分别展示 5 小时和 1 周用量窗口。
- Access Token 临近过期时才使用 Refresh Token 刷新；没有 Refresh Token 的凭证在 Access Token 有效期内保持不变，失效后需要重新导入。
- Codex 模型探测会汇总端点内各 Key 的可用模型；真实请求只在规则候选池中进行故障切换。
- Codex Responses 保持 SSE，上游失败事件会进入正常熔断与候选切换流程。

详细说明见 [Codex OAuth Provider](codex-provider.md)。

## 多协议规则组

- 规则通过多选入口同时支持 `chat`、`response`、`codex`、`message`、`claude_code` 和 `gemini`。
- 同一规则组内的规则不能选择重叠入口，避免路由含义不确定。
- 内置 `default` 固定支持全部入口，并动态使用全部启用 Key。
- Provider 和入口协议严格匹配，不会为了兜底跨到不兼容协议。
- 鉴权、额度、限流、模型不支持及常见上游错误，只会切换到当前规则候选池中的下一个兼容 Key。

## Agent 统一网络出口

- 未绑定 Agent 的端点由主服务直接访问上游。
- 绑定 Agent 后，模型列表、自动/手动探测、API Key 测试、OAuth/Codex Token 刷新和真实推理请求全部使用该 Agent。
- Agent 不可用时不会静默回退主服务直连，而是按错误、熔断和规则候选切换处理。
- Codex 默认需要在 Agent allowlist 中同时允许 `chatgpt.com` 和 `auth.openai.com`。

## 控制台与安全

- API 端点、规则入口和 Provider 计数显示已同步新语义。
- 访客模式不再暴露真实上游端点地址。
- Codex Auth JSON 只保留运行所需字段并加密存储，不向下游透传账号额度与 Cookie header。
- 管理员可以按 Key 查看 Codex 用量，不再把端点内第一个 Auth 的用量当作整体用量。

## 数据库与部署

- SQLite 继续适合本地和低并发单机使用。
- 生产环境建议使用 PostgreSQL 与 Redis。
- 项目按新部署方式运维：空数据库按当前 ORM 模型创建 v0.2 schema，不承诺自动升级任意旧版本数据库。
- 升级已有实例前先备份数据库与 `LLM_DATA_ENCRYPTION_KEY`，再自行执行必要 SQL，或切换到新的空数据库。

## v0.2 边界

- Codex OAuth 仍是隔离的实验 provider，需要自行评估账号权限、服务条款和风控风险。
- 不做 OpenAI、Anthropic、Gemini 之间的跨 provider 协议翻译。
- Agent 是可信执行节点，生产环境必须配置明确的目标 allowlist。
